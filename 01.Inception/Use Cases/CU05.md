## CU05 – Reportar un prompt y moderarlo como admin

**Nombre**  
CU05 – Reportar y moderar prompt

**Actores principales**  
- Usuario autenticado (rol `user` o `admin`) que reporta.  
- Administrador (`admin`) que revisa y decide.

**Objetivo**  
Permitir que los usuarios reporten prompts problemáticos y que los admins los revisen, pudiendo borrarlos y aplicar penalización de XP al autor.

**Precondiciones**

- Usuario autenticado en la app web.  
- El prompt es visible (público/compartido).  
- Existen uno o más usuarios con rol `admin`.  

***

### Parte A – Reportar un prompt (usuario)

**Flujo principal (reportar)**

1. El usuario abre el detalle de un prompt o lo ve en un listado.  
2. Junto al prompt ve una opción “Reportar” (icono o botón).  
3. El usuario pulsa “Reportar”.  
4. Se abre un pequeño formulario modal con:
   - Selector de motivo (spam, contenido ofensivo, ilegal, otro).  
   - Campo opcional de comentario.  
5. El usuario elige motivo, añade (opcional) comentario y pulsa “Enviar reporte”.  
6. El frontend envía `POST /api/prompts/{id}/report` con motivo y comentario.  
7. El backend crea una entrada en `prompt_reports` con:
   - `prompt_id`, `reporter_id`, `reason`, `comment`, `status = "open"`, `created_at`.  
8. La API devuelve éxito.  
9. La UI muestra un mensaje tipo “Gracias, revisaremos este contenido”.

**Flujos alternativos / errores**

- 6A. El mismo usuario ya ha reportado este prompt y el reporte sigue `open`  
  - El backend puede devolver un error controlado o simplemente no crear un segundo reporte; la UI muestra “Ya has reportado este prompt”.  
- 4A. El usuario cancela el formulario  
  - No se envía nada, no se crea reporte.

***

### Parte B – Revisar y moderar el prompt (admin)

**Flujo principal (revisión y borrado con penalización)**

1. Un admin accede a la sección `/admin/reportes` en la app web.  
2. Ve una lista de reportes con estado `open`, con columnas:
   - Prompt (título / extracto).  
   - Autor del prompt.  
   - Número de reportes que tiene ese prompt.  
   - Motivo principal.  
   - Fecha del reporte. [dev](https://dev.to/kyleledbetter/supabase-dashboards-that-dont-suck-production-ready-analytics-without-the-setup-hell-330f)
3. El admin selecciona un reporte para revisarlo (detalle del reporte).  
4. La pantalla de detalle muestra:
   - Texto completo del prompt.  
   - Autor del prompt y sus métricas básicas (XP, nº de prompts, likes, etc.).  
   - Lista de todos los reportes asociados a ese prompt (posiblemente varios usuarios).  
5. El admin decide entre tres acciones:
   - “Mantener” (rechazar reporte).  
   - “Ocultar de la comunidad” (opcional, lo hace no público).  
   - “Borrar prompt” (por incumplimiento claro).  
6. Si el admin elige “Borrar prompt”:
   - El frontend llama a `POST /api/admin/prompts/{id}/delete`.  
7. El backend:
   - Marca el prompt como borrado o lo elimina físicamente (según la estrategia).  
   - Actualiza todos los reportes asociados a ese prompt a `status = "resolved"` o similar.  
   - Crea un evento negativo en `xp_events` para el autor del prompt:
     - `type = "prompt_deleted_due_to_report"`, `amount = -XP_penalizacion`.  
8. La API devuelve éxito.  
9. La interfaz admin actualiza la lista de reportes (ya no muestra ese prompt como pendiente) y puede mostrar una notificación “Prompt borrado y XP penalizado”.

**Flujo alternativo – Mantener prompt (rechazar reporte)**

5B. El admin elige “Mantener”:

1. El frontend llama a `POST /api/admin/prompts/{id}/keep`.  
2. El backend marca los reportes relacionados como `status = "rejected"` (o similar), sin tocar el prompt ni el XP.  
3. La UI actualiza la lista de reportes, quitándolos de `open`.

**Flujos alternativos / errores**

- 7A. Varios admins actúan al mismo tiempo sobre el mismo prompt  
  - Debes controlar que una segunda acción detecte que el prompt ya fue borrado o marcado y devuelva un error controlado (“Ya se ha resuelto este reporte”).  
- 6A. Error de autenticación o autorización  
  - Si el usuario no es admin, la API devuelve 403; la UI muestra “No tienes permisos para moderar”.
